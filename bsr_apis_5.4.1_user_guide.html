Host configurations

Network settings

The maximum socket receive buffer size needs to be set to 67108864 (64 MB). (recommended for field PCs with many sensors) The minimum requirement socket receive buffer size should be to be set to 33554432 (32 MB). (recommended for lab users)

Temporary configuration:

  1. run sudo sysctl -w net.core.rmem_max=67108864.

Permanent configuration:

  1. add net.core.rmem_max=67108864 line into /etc/sysctl.conf file.
  2. add net.core.netdev_max_backlog=30000 line into /etc/sysctl.conf file.
  3. run sudo sysctl -p.

Note: To print buffer size, run the following command: cat /proc/sys/net/core/rmem_max, or cat /proc/sys/net/core/netdev_max_backlog

Secured radar

Use the is_secured flag when connecting to a radar that requires authenticated (secured) access. Set is_secured=True

Parameter Type Default Description
is_secured bool False Set to True when the radar uses secured connection.

Example — defining a secured radar (e.g. as in Secured_local.py):

from bsr_apis import Radar, OperationMode

with Radar(
    ip='192.168.1.31',
    is_secured=True,
) as radar:
    if radar.update_fw(force=True):
        connect_response = radar.connect()
        if connect_response:
            # use radar (e.g. set_operation_mode, start_recording, ...)
            pass

Basic APIs

Ping

# Send ping request and wait for response using default timeout of 5 [sec]
# ping is either a2a ping req or regular os based ping, depanding if radar is connected or not.

if ping_response := radar.ping():
    # ping response contains ping elapsed time [sec]
    ping_elapsed_time = ping_response.elapsed_time  

# Send ping request and wait for response using user defined timeout [sec]
if ping_response := radar.ping(timeout=2):  # Example
    # ping response contains ping elapsed time [sec]
    ping_elapsed_time = ping_response.elapsed_time

Reset

# There are 2 reset types: COLD and RMGR

# Send COLD reset request and wait for wait_for_activations (depanding on radar state)
if reset_response := radar.reset(ResetType.COLD):
    # reset response contains reset elapsed time [sec]
    reset_elapsed_time = reset_response.elapsed_time

# Send COLD reset request and return without waiting for activations
if reset_response := radar.reset(ResetType.COLD, wait_for_activations=False):
    # reset response contains reset elapsed time [sec]
    reset_elapsed_time = reset_response.elapsed_time

Set State

# There are currently 3 supported states to set: STANDBY, SCANNING and FW_UPDATE
# If the radar is already in the requested state, returned response will be True
# Be sure to take radar state constrains into consideration

# Send set state and wait for activations (depanding on radar state)
if set_state_response := radar.set_state(RadarState.SCANNING):
    # your code here

# Send set state and return without waiting for activations
if set_state_response := radar.set_state(RadarState.STANDBY, wait_for_activations=False):
    # your code here

FW update should be performed in following cases:

  1. Before calling radar.connect()
  2. If radar.connect() failed with response status FW_VERSION_MISSMATCH

FW update will fail in following cases:

  1. Current FW version is earlier than 3.0.0 (Officially supported FW version: 3.6.3 and later)
  2. Selected configuration not supported
  3. Current configuration not supported
  4. Device type not available
  5. Device type not supported
  6. Recording is running

Note: FW update will be skipped if supported FW version already installed (unless forced)
If a version lower than v3.14.5-0-gfaf665e3, the FW update will be performed to v3.14.5-0-gfaf665e3 using the appropriate fw_updater to requested Modality in New version without changing Modality or IP for the intermediate version).

IMPORTANT:
- If you need to apply specific configuration as shown in snippets below you must perform power cycle manually in following steps:
1) Once you see ping logs with severity WARNING, set media convertor to required state.
2) Turn OFF power supply.
3) Wait 5 seconds.
4) Turn ON power supply.
4) Wait for the FW update to complete (ping logs with severity WARNING will still appear). 

# Get supoprted FW version as string
supported_fw_version = radar.supported_fw_version

# Update FW by automatically detect current configuration
status = radar.update_fw()

# Update FW using modality configurations
status = radar.update_fw(FWUpdateModality.DEFAULT) #Default preset = Lab_Val 
status = radar.update_fw(FWUpdateModality.AVKIT58) #Default preset = Mobileye 
status = radar.update_fw(FWUpdateModality.DR64) #Default preset = Mobileye 
status = radar.update_fw(FWUpdateModality.PROD) #Default preset = SW_Default 

# Update FW using preset predefined congiration (example)
#BSR32 -> BSR32_FRONT_CENTER_BOTTOM / BSRC -> BSRC_FRONT_RIGHT_BOTTOM
status = radar.update_fw(FWUpdateModality.DEFAULT.Field_Val) 
status = radar.update_fw(FWUpdateModality.AVKIT58.Mobileye) 
status = radar.update_fw(FWUpdateModality.DR64.LARI)

# Update FW using explicit predefined congiration (example)
status = radar.update_fw(FWUpdateModality.DEFAULT.Field_Val.BSR32_FRONT_CENTER_BOTTOM)
status = radar.update_fw(FWUpdateModality.AVKIT58.Mobileye.BSRC_REAR_LEFT_BOTTOM)
status = radar.update_fw(FWUpdateModality.DR64.LARI.BSRC_FRONT_LEFT_BOTTOM)


# To perform FW update again please pass "force" flag
status = radar.update_fw(force=True)

# Update FW to specific operation mode (example)
status = radar.update_fw(operation_mode=OperationMode.PC1_RDC_10FPS_MRR_LR_76128)

Managed RFIC Register Operations

BSR APIs provides advanced functionality for managing RFIC register operations through the managed registers interface. This allows you to write, read, and manage RFIC register entries with automatic memory tracking.

Overview

The managed RFIC functions work together as a system: - managed_registers.write_req() - Writes RFIC entries to radar memory - managed_registers.read_req() - Reads RFIC entries from radar memory
- managed_registers.remove_all_req() - Removes all tracked entries from radar memory

The API automatically tracks which entries have been added and manages configuration IDs.

Basic Workflow

from bsr_apis import Radar, ManagedRegisters
from bsr_apis import ManagedRegistersTarget, CalibTiming, RficUnit
import time

# calc address with dictionary :
# chip_num_dict = {'RX0': 0x12,
#                 'RX1': 0x13,
#                 'RX2': 0x14,
#                 'RX3': 0x15,
#                 'RX4': 0x16,
#                 'RX5': 0x17,
#                 'RX6': 0x18,
#                 'RX7': 0x19,
#                 'TX0': 0x1a,
#                 'TX1': 0x1b,
#                 'TX3': 0x1d,
#                 'TX4': 0x1e}
# new_address = (chip_num_dict[Chip] <<24) + address

# Create RFIC entries (example)
entries = [
    RficUnit(address=0x1aED008C, value=0x00123456, mask=0xffffffff),
    RficUnit(address=0x12ED008C, value=0x00654321, mask=0xffffffff),
]

# Step 1: Write entries to radar memory
write_status = radar.managed_registers.write_req(
    target=ManagedRegistersTarget.RX4.value,
    timing=CalibTiming.PRE_RTC_SYNC_PHASE2.value,
    entries=entries
)

# Step 2: Read the same entries from radar memory
read_status = radar.managed_registers.read_req(
    target=ManagedRegistersTarget.RX4.value,
    timing=CalibTiming.PRE_RTC_SYNC_PHASE2.value,
    entries=entries
)

# data will be an np.array with the following fields : 
    # frame_id
    # timestamp_sec
    # timestamp_nsec
    # sensor_id
    # sensor_type
    # scan_mode
    # reserved_0
    # frequency
    # num_entries
    # report_type
    # entries  # array with max of 10 entries

if read_status:
    data = radar.get_managed_rfic()
    pp(f"\n\tRead data: {[hex(dd) for dd in data['entries']['data']]}") 

# Step 3: Sleep before cleanup (important for radar stability)
time.sleep(1)  # Allow radar to process

# Step 4: Remove all entries from radar memory
remove_read_status = radar.managed_registers.remove_all_req(
    target=ManagedRegistersTarget.RX4.value,
    timing=CalibTiming.PRE_RTC_SYNC_PHASE2.value,
    is_write=False  # Remove read entries
)

remove_write_status = radar.managed_registers.remove_all_req(
    target=ManagedRegistersTarget.RX4.value,
    timing=CalibTiming.PRE_RTC_SYNC_PHASE2.value,
    is_write=True  # Remove write entries
)

# Step 5: Sleep after cleanup (important for radar stability)
time.sleep(1)  # Allow radar to clean up

# Now you can set new data for next time

Reading Without Adding to Memory

To read entries without adding them to the internal tracking lists, pass an empty list:

# This will read from radar memory but won't track the entries internally
read_status = radar.managed_registers.read_req(
    target=ManagedRegistersTarget.RX4.value,
    timing=CalibTiming.PRE_RTC_SYNC_PHASE2.value,
    entries=[]  # Empty list - no entries added to internal tracking
)

Important Notes

  1. Sleep Requirements: Always include sleep periods:
    • Before managed_registers.remove_all_req(): Allow radar to process pending operations
    • After managed_registers.remove_all_req(): Allow radar to clean up memory
  2. Memory Tracking: The API maintains internal lists of entries and automatically manages configuration IDs
    • Duplicate entries are automatically filtered out
    • The config ID is accessible via managed_registers.config_id
  3. Return Status: All functions return a2a_debug_prod.DebugProductionRespStatus:
    • OK - Operation successful
    • Other values indicate errors

EEPROM

BSR APIs supports reading and writing EEPROM data for both BPU and RF boards.

Availability: radar.bpu_eeprom and radar.rf_eeprom are available after either: - radar.connect() returns successfully, or - radar.prepare_for_eeprom() returns True (without full connect).

If neither has been done, both properties are None. Radar object must be connected (or prepared for EEPROM) before doing any EEPROM read/write.

Prepare for EEPROM

If the user wants to use EEPROM without connecting and burning the radar, call prepare_for_eeprom() to initialize the A2A module required for EEPROM communication.

from bsr_apis import Radar

with Radar(ip='192.168.1.31') as radar:
    # Prepare radar for EEPROM operations
    # Returns True if successful, False otherwise
    if radar.prepare_for_eeprom():
        # Now you can use rf_eeprom and bpu_eeprom operations
        bpu_data = radar.bpu_eeprom.read()
        rf_data = radar.rf_eeprom.read()

        if bpu_data is not None:
            print(f"BPU Serial: {bpu_data.bpu_serial_number}")
        if rf_data is not None:
            print(f"RF Serial: {rf_data.rf_serial_number}")
    else:
        print("Failed to prepare for EEPROM operations")

Note: This function initializes the A2A module if not already initialized. After it returns True, you can use all radar.bpu_eeprom and radar.rf_eeprom methods.

Read EEPROM

# Read EEPROM data from BPU board
# Returns BPUEepromFields instance or None if failed
bpu_eeprom_data = radar.bpu_eeprom.read()

# Read EEPROM data from RF board  
# Returns RFEepromFields instance or None if failed
rf_eeprom_data = radar.rf_eeprom.read()

Write EEPROM

from bsr_apis import BPUEepromFields, RFEepromFields

# Write EEPROM data to BPU board
# Pass a BPUEepromFields instance with the fields to write
fields = BPUEepromFields(
    version=1,
    bpu_serial_number="00112233445566778899AABBCCDDEEFF",
    # ... other fields as needed ...
)
status = radar.bpu_eeprom.write(fields)  # Returns True if successful, False otherwise

# Write EEPROM data to RF board
# Pass a RFEepromFields instance with the fields to write
rf_fields = RFEepromFields(
    version=1,
    rf_serial_number="00112233445566778899AABBCCDDEEFF",
    # ... other fields as needed ...
)
status = radar.rf_eeprom.write(rf_fields)  # Returns True if successful, False otherwise

Partial write: Only fields that are not None are written. Other EEPROM bytes are left unchanged. To update a single field, read the current data, set the desired field, then write.

Error handling: read() returns None on failure (check logger for details). write() returns False on validation error or transport failure; validation errors are logged. Always check return values and back up EEPROM data before writing.

Note: EEPROM operations should be performed carefully as incorrect data can affect radar functionality. Always backup existing EEPROM data before writing new data.

Backup and restore

  • NumPy: Use np.save() / np.load(..., allow_pickle=True).item() as in the example above to store and reload full BPUEepromFields or RFEepromFields instances.
  • Raw bytes: For BPU, bpu_data.to_bytes() returns the EEPROM layout as bytes; restore with BPUEepromFields.from_bytes(raw_data). Similarly, RF: rf_data.to_bytes() and RFEepromFields.from_bytes(raw_data).

If connect fails with ConnectResponseStatus.EEPROM_FAILED_GET_LOCATION, EEPROM read/write may be unavailable until the underlying issue (e.g. hardware or FW) is resolved.

FAZIT Number Generation Example

The FAZIT number is a 26-character string used for EEPROM burning. To generate a valid FAZIT string for the fazit field in BPUEepromFields, use the create_fazit_number function from api_utils.common. For EEPROM, use the string excluding the first 3 header bytes.

Example

from api_utils.common import create_fazit_number

# Example values:
plant_num1 = "A5G"         # First part of plant number (up to 3 chars)
plant_num2 = "001"         # Second part of plant number (up to 3 chars)
day = 17                   # Manufacturing day
month = 5                  # Manufacturing month
year = 2023                # Manufacturing year
prod_test_sys_number = "1234"  # Up to 4 chars
supplier_num = "5678"          # Up to 4 chars

# Generate the full FAZIT string (26 chars, includes 3 header bytes)
fazit_number = create_fazit_number(
    plant_num1,
    plant_num2,
    day,
    month,
    year,
    prod_test_sys_number,
    supplier_num
)

# Use in BPUEepromFields:
from bsr_apis import BPUEepromFields
fields = BPUEepromFields(
    # ... other fields ...
    fazit=fazit_number,
)
status = radar.bpu_eeprom.write(fields)
if status:
    print("EEPROM write successful")
else:
    print("EEPROM write failed")

Parameter Reference: - plant_num1: First part of plant number (1-3 chars, will be zero-padded) - plant_num2: Second part of plant number (1-3 chars, will be zero-padded) - day, month, year: Manufacturing date (day/month/year as integers) - prod_test_sys_number: Production test system number (up to 4 chars) - supplier_num: Supplier number (up to 4 chars)

Note: The returned string is latin-1 encoded and each character represents a byte.
